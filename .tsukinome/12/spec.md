# Spec — Issue #12: Add formatDuration(ms) utility for human-readable duration strings

**Type:** feature

> Generated by Tsukinome. This file is the source of truth for the work; edits here drive the build.

## Summary

Add a self-contained utility function `formatDuration(ms: number): string` that converts a millisecond count into a compact, human-readable duration string (e.g. "820ms", "5s", "1m 5s", "2h 30m"). It selects the largest sensible unit, rounds to the nearest whole unit with carry-across behavior at unit boundaries, and handles non-finite and non-positive input by returning "0ms". The function lives in its own module with unit tests and is not wired into any existing code.

## Requirements

Each requirement is tagged with a confidence level (explicit / inferred / assumption / unknown).

- **R1** `[explicit]` The function must be exported as `formatDuration(ms: number): string` from its own dedicated utility module.
- **R2** `[explicit]` Non-finite input (NaN, Infinity, -Infinity) must return "0ms".
- **R3** `[explicit]` Input where ms <= 0 (including negatives and -0) must return "0ms".
- **R4** `[explicit]` For 0 < ms < 1000, the result must be whole milliseconds rounded to the nearest integer, formatted as "{ms}ms".
- **R5** `[explicit]` For 1000 <= ms < 60000, the result must be whole seconds formatted as "{s}s".
- **R6** `[explicit]` For 60000 <= ms < 3600000, the result must show minutes and seconds always, formatted as "{m}m {s}s" (e.g. 60000 -> "1m 0s").
- **R7** `[explicit]` For ms >= 3600000, the result must show hours and minutes, dropping seconds, formatted as "{h}h {m}m".
- **R8** `[explicit]` Rounding must be to the nearest whole unit (not truncation) and must carry across unit boundaries (e.g. 59999 -> "1m 0s", 3599999 -> "1h 0m").
- **R9** `[explicit]` The utility must be covered by unit tests, including the examples listed in the issue.
- **R10** `[explicit]` The function must not be integrated into any existing caller and must introduce no new dependencies.
- **R11** `[inferred]` When rounding causes a carry that crosses the unit-selection threshold, the higher unit's formatting rules are applied (e.g. a value that rounds up to a full hour is displayed in hours/minutes form).

## Non-Goals

- Integrating formatDuration into the cost summary or any other caller.
- Adding any new dependencies.
- Supporting units larger than hours (e.g. days) or localization/pluralization variations.

## Edge Cases

- Sub-millisecond fractional input rounding to the nearest whole ms (e.g. 820.6 -> "821ms").
- Values just below a boundary that round up and carry to the next unit (59999 -> "1m 0s", 3599999 -> "1h 0m").
- Negative zero (-0) treated as non-positive -> "0ms".
- Non-finite inputs NaN, Infinity, -Infinity all -> "0ms".
- Exact boundary values (1000, 60000, 3600000) selecting the correct unit.
- Very large finite ms values (e.g. tens of hours) still formatting as hours and minutes.

## Assumptions

- When a rounding carry pushes a value across the unit-selection threshold, the display uses the higher unit's formatting rules; this is implied by the carry examples but not stated as a general rule.
